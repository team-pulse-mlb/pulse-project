# 추천 점수 계산 방법

## 1. 추천 점수의 사용처

| 사용처 | 설명 |
|---|---|
| 홈 추천 정렬 | 진행 중 경기를 `watch_score` 순으로 정렬한다. |
| 예정 경기 정렬 | 예정 경기를 `pregame_score` 순으로 정렬한다. |
| 종료 경기 정렬 | 종료 경기를 라이브 중 기록된 최고 점수(`peak_base_score`) 순으로 정렬한다. |
| 알림 판단 | 임계 진입(85)과 급등 조건을 알림 트리거로 사용한다. |
| 경기 전환 안내 | 현재 보는 경기보다 점수가 높은 경기를 안내한다. |
| 다시 보기 구간 | 라이브 중 점수가 높았던 구간을 구간 단위로 저장한다. |

## 2. 핵심 원칙

| 항목 | 내용 |
|---|---|
| 전 경기 동일 수집 | 진행 중 경기 전체에서 같은 데이터를 같은 주기로 수집한다. 상위 N개 경기만 정밀 분석하지 않는다. |
| 라이브 1회 계산 | 분석은 경기가 진행 중일 때 실시간으로 1회만 수행하고, 결과와 타임스탬프를 저장한다. 경기 종료 시 재분석하지 않고 상태만 진행 중 → 종료로 변경한다. |
| 랭킹 신호와 해석 신호의 분리 | 랭킹에는 전 경기에서 동일하게 산출되는 견고한 신호만 넣는다. 노이즈가 있거나 정규화가 어려운 신호(투수 흔들림, 강한 타구)는 추천 이유 태그와 상세 화면 표시에만 사용한다. |
| 개인화 | 랭킹은 전체 사용자 공용 1개만 유지한다. 관심 팀/선수 가산은 API 조회 시점에 적용한다. |
| 상수 관리 | 가중치와 임계값은 `backend/src/main/resources/scoring.yml`에서 관리한다. 변경 시 `version`을 올린다. 현행 `scoring.yml`(version 1)은 이 문서 확정 이전의 리플레이 프로토타입 상수이므로, scorer 구현 착수 전 이 문서 기준으로 version 2 갱신을 선행한다. |

## 3. 점수 체계

| 경기 상태 | 점수 | 정렬 기준 | 계산 시점 |
|---|---|---|---|
| 예정 | `pregame_score` (0~100) | 추천순 | 선발·배당·순위 데이터 갱신 시 |
| 진행 중 | `watch_score` (0~100) | 추천순 | 라이브 폴링 사이클마다 |
| 종료 | `peak_base_score` · `replay_segments` | 추천순 | 라이브 중 계산한 값을 그대로 사용 |

## 4. 진행 중 경기 — watch_score

### 4.1 공식

```text
1. 기본 점수 계산
base_score =
    후반/연장 가산
  + 점수 차 가산
  + 최근 득점 가산
  + 리드 변경 가산
  + 빅이닝 가산
  + 압박 가산
  + 카운트/아웃 가산
  + 초반 난타 가산

2. 경기 중요도 보정 적용
adjusted_score = base_score × 경기 중요도 보정

3. 사전 가산 적용
raw_watch_score = adjusted_score + 사전 가산

4. 최종 점수 제한
watch_score = clamp(raw_watch_score, 0, 100)

최종 점수는 항상 0점 이상 100점 이하로 제한한다.
```

### 4.2 신호 정의

| 신호 | 조건 | 가산점 | 데이터 |
|---|---|---|---|
| 후반/연장 | 7~8회 / 9회 / 연장 | +6 / +12 / +18 | `/games.period` |
| 점수 차 | 0~1점 / 2점 / 3점 / 4점 이상 | +15 / +9 / +3 / +0 | `/games` 팀별 runs |
| 최근 득점 | min(득점×6, 15) × 접전 배율 × exp(-경과초/180) | 최대 +30 (동점 시) | `/plays.scoring_play`, `score_value`, observed_at |
| 리드 변경 | 앞선 팀이 바뀐 뒤 +9 × exp(-경과초/300) | 최대 +9 | `/plays.home_score`, `away_score`, observed_at |
| 빅이닝 | 현재 이닝에 득점 play 2개 이상 | +9 | `/plays.scoring_play`, `inning` |
| 압박 | 현재 타석 만루 / 2루 또는 3루 주자 | +6 / +4 | `ScoreTask.situation` (원천 `/plate_appearances.runner_on_*`) |
| 카운트/아웃 | 풀카운트 +3, 2아웃 +3 (합산 최대 +5) | +3~+5 | `ScoreTask.situation` (원천 `/plays.balls`, `strikes`, `outs`) |
| 초반 난타 | 1~3회이고 양 팀 득점 합 7점 이상 | +5 | `/games` 이닝별 점수 |


\* 접전 배율: 득점이 발생한 직후의 **점수 차**가 작을수록 더 높은 배율을 적용한다.

| 득점 직후 점수 차 | 배율 |
|---|---|
| 동점 | ×2.0 |
| 1점 차 | ×1.5 |
| 2점 차 | ×1.2 |
| 3점 차 이상 | ×1.0 |

압박·카운트 신호는 poller가 `ScoreTask.situation`으로 전달한 현재 타석 스칼라(만루·득점권 boolean, 아웃·볼·스트라이크)에서 산출한다. `/plate_appearances`는 운영 Postgres에 영속하지 않으므로 scorer는 이 스칼라를 사용한다. 전 라이브 경기에서 동일 주기로 수집되고 boolean으로 판정할 수 있으므로 랭킹에 포함한다. 투수 흔들림, 강한 타구, 관심 선수 타석 같은 상세 신호는 정렬 점수에 넣지 않고 태그와 상세 화면 표시에만 사용한다.

위 가중치는 신호 최댓값 합산에 경기 중요도 보정·사전 가산을 더한 현실적 최댓값이 100 부근이 되도록 잡은 초기값이며, `scoring.yml`에서 관리하고 백테스트로 확정한다.


### 4.3 시간 감쇠

이벤트성 신호(최근 득점, 리드 변경)는 폴러가 기록한 최초 관측 시각(observed_at) 기준으로 지수 감쇠한다. 폴링 주기가 감쇠 계산의 오차 상한이 된다.

### 4.4 경기 중요도 보정

`/standings` 일 배치 데이터로 판정한다. 경쟁권은 `playoff_percent`(플레이오프 진출 확률) 10~90%로 정의한다.

| 조건 | 배수 |
|---|---|
| 포스트시즌 경기 | ×1.15 |
| 양 팀 모두 경쟁권 | ×1.10 |
| 한 팀만 경쟁권 | ×1.05 |
| 기본 | ×1.00 |
| 양 팀 모두 `playoff_percent` < 10% | ×0.90 |

곱셈 보정이므로 조용한 경기에는 거의 영향이 없고, 라이브 신호가 큰 경기만 증폭된다. 보정이 라이브 신호를 넘어서지 않는다는 원칙을 구조로 보장한다.

### 4.5 사전 가산과 개인화

- 사전 가산: `pregame_score / 10` (0~10점). 경기 초반 라이브 신호가 쌓이기 전의 정렬 공백을 메운다. 경기 내내 상수로 유지한다.
- 관심 팀/선수 가산: 최대 +15. Redis 공용 랭킹을 읽은 뒤 API 응답 조립 시점에 사용자별로 더한다.

## 5. 예정 경기 — pregame_score

```text
pregame_score = 접전 기대(0~30) + 선발 매치업(0~40) + 순위 경쟁(0~30)
```

| 구성 | 공식 | 데이터 |
|---|---|---|
| 접전 기대 | 기본: 30 × max(0, 1 − abs(양 팀 승률 차) / 0.15). 배당 스냅샷 확보 시 `PREGAME_FINAL`을 우선 사용하고, 없으면 `FIRST_SEEN`을 사용해 벤더 중앙값 moneyline 암시 승률 p로 30 × max(0, 1 − abs(p − 0.5) / 0.25) 대체. 둘 다 없으면 승률 차 폴백 유지 | `/standings.win_percent`, `/odds.moneyline_*` |
| 선발 매치업 | 선발별 20 × clamp((4.6 − 시즌 ERA) / 2.0, 0, 1)의 합. 선발 미확정 시 해당 선발 0점 | `/lineups.is_probable_pitcher`, `/season_stats.pitching_era` |
| 순위 경쟁 | 양 팀 경쟁권 30 / 한 팀 15 / 그 외 0 (4.4와 같은 경쟁권 판정) | `/standings.playoff_percent` |

- 배당은 당일 슬레이트만 제공되므로, 전일에는 승률 차 기반으로 계산하고 당일 첫 관측(`FIRST_SEEN`) 등장 시점부터 시장 기반으로 대체하며, 시작 직전 `PREGAME_FINAL` 확보 시 이를 우선한다.
- 선발 `season_stats`는 pregame 계산이 저빈도·비실시간이므로 계산 시점에 확정 선발(`is_probable_pitcher`)의 `player_ids[]`로 온디맨드 조회하거나 일 배치로 `player_season_stats`에 적재한다. 별도 P1 테이블 승격은 하지 않는다. 조회 실패 시 직전 캐시값을 사용하고, 값이 없으면 해당 선발 0점으로 처리한다.

- 배당은 내부 계산 전용이며 UI에 노출하지 않는다. 예정 경기의 추천 이유 태그는 순위 경쟁·선발 매치업·관심 팀에서만 생성한다.
- 관심 팀/선수 가산은 진행 중 경기와 동일하게 조회 시점 +15로 적용한다.
- 계산 경로: poller가 경기 전 입력 갱신 시 `lifecycleState=PREGAME`인 ScoreTask를 발행하고, scorer가 DB의 `lineups`·`odds_snapshots`·`standings`·`player_season_stats`를 읽어 계산한 뒤 `games.pregame_score`·`pregame_inputs`에 저장한다. 외부 API 온디맨드 조회는 poller가 task 발행 전에 끝낸다.
- 예정 경기의 추천 이유 태그는 저장하지 않고, 조회 시점에 `pregame_inputs`(순위 경쟁·선발 매치업)와 관심 팀에서 파생한다.

## 6. 종료 경기 — peak_base_score와 replay_segments

종료 경기는 라이브 중 저장한 분석 결과를 그대로 사용한다. 종료 시점에 재분석하지 않는다.

### 6.1 다시 보기 구간 (라이브 중 확정)

스코어러는 라이브 계산 중 다음 규칙으로 구간을 실시간 관리한다.

1. `base_score`가 60 이상으로 올라가면 구간을 연다 (시작 play order, 이닝 기록).
2. `base_score`가 50 미만으로 내려가면 구간을 닫는다 (히스테리시스로 경계 진동 방지).
3. 직전 구간과 1 하프이닝 이내면 하나로 병합한다.
4. 구간에는 play order 범위, 이닝 범위, 구간 내 최고 점수, 구간 중 발생한 태그를 저장한다.
5. 경기 종료 감지 시 열려 있는 구간을 닫고 상태만 종료로 변경한다.

노출은 구간 내 최고 점수 기준 상위 3개로 한다. 보호 모드에서는 최종 점수와 승패를 드러내지 않는 문구만 사용한다.

구간 임계(60/50)는 초기 가중치 기준값이며, 백테스트 구간 커버리지 지표로 재점검한다.


### 6.2 종료 경기 정렬

- 정렬 키는 라이브 중 갱신한 `peak_base_score`(경기 최고 base_score)다. 중요도 보정과 사전 가산을 제외한 값이라 경기 자체의 긴장도만 반영한다.
- 카드에는 구간 개수 배지를 표시할 수 있다. 점수·승패는 노출하지 않는다.
- 라이브 수집 이전에 종료된 경기는 분석 결과가 없으므로 0점으로 처리해 하단에 정렬한다.

## 7. 결측치 처리 규칙

- 신호 계산에 필요한 필드가 없으면 해당 신호는 0점으로 처리하고 계산을 계속한다. 예외를 던지지 않는다.
- 집계 지연으로 `null`이 올 수 있는 필드(`expected_woba` 등)는 소비 측에서 null 방어를 기본으로 한다.
- 외부 API 오류 시 마지막 정상 점수를 유지하고 오래된 데이터임을 표시한다.

## 8. 백테스트 계획

하이라이트 구간, 사람 평가 점수, 수작업 라벨 같은 추가 데이터는 사용하지 않는다. 모든 지표는 API 데이터에서 자동 산출한다.

### 8.1 데이터와 재생 방법

- 데이터: DB 이전 완료 후 백테스트는 **운영 DB에 적재된 이력**으로 수행한다. 입력은 `plays`·`games`·`watch_scores`·`odds_snapshots`·`standings`이며, S3에서 이전·보존한 과거 데이터를 포함한다. S3 원본을 직접 재생하는 방식은 DB 이전 전까지의 잠정 방식이며, 이전 완료 후에는 사용하지 않는다.
- 데이터 구분: 이전 데이터는 `source` 컬럼(`S3_LIVE_ARCHIVE`·`S3_BACKFILL`)과 이전 경계 시각으로 운영 수집분(`OPERATIONAL`)과 구분한다. 재생 로직은 `source`에 따라 시간 감쇠 방식을 분기한다.
- 재생: 경기별 plays를 order 순으로 점수 함수에 주입한다.
  - `OPERATIONAL`·`S3_LIVE_ARCHIVE`(observed_at 실측 보유): 실제 시간 감쇠로 재생한다. `S3_LIVE_ARCHIVE`는 관측 주기가 운영(약 20초)과 다를 수 있어 감쇠 오차 상한이 커진다.
  - `S3_BACKFILL`(과거 시즌, 벽시계 시각 없음): 시간 감쇠 항을 order 윈도우로 근사한다(최근 득점: 이후 15 plays 선형 감쇠, 리드 변경: 이후 25 plays 선형 감쇠).
- 압박 신호: 타석 시작 시 주자 상태가 `plays.runner_on_*`로 영속되므로 DB 기반 재생에서 재계산·가중치 튜닝이 가능하다. `runner_on_*`가 `null`인 play(PA 미대응)는 해당 시점 압박을 0점 처리한다.
- 그 외 PA 기반 상세 신호(강한 타구·구속 저하·긴 타석): 랭킹 신호가 아니라 태그·이벤트 표시 전용이므로 가중치 튜닝 대상이 아니다. PA 원본은 운영 DB에 영속하지 않으므로 DB만으로는 재산출할 수 없고, `game_events`에 저장된 추출 결과를 사용하며 소급 재추출이 필요하면 S3 PA 아카이브를 재생한다.

### 8.2 검증 지표

| 지표 | 방법 | 목적 |
|---|---|---|
| 정렬 품질 | 경기 단위 접전 지표(최종 점수 차, 리드 변경 횟수, 8회 이후 동점 여부, 연장 여부)와 경기 peak `watch_score`의 순위 상관 | 점수 상위 경기가 실제 접전이었는지 |
| 예측력 | 시점 t의 `base_score`와 "이후 12 plays 내 득점 또는 리드 변경 발생"의 판별력(AUC) | 점수가 긴장을 결과보다 먼저 잡는지 |
| 알림 빈도 | 임계 85 기준 하루 알림 발생 수 분포 | 알림이 하루 2~5건 범위가 되도록 임계·상수 조정 |
| 구간 커버리지 | replay 구간이 득점·리드 변경 play를 포함하는 비율 대비 구간 총 길이 | 구간 임계 60/50의 적정성 |

### 8.3 튜닝 절차

`scoring.yml` 상수를 그리드 서치로 조정한다. 지표 우선순위는 예측력 > 정렬 품질 > 알림 빈도로 한다. 상수 변경 시 `version`을 올리고 `tune:` PR에 영향 리포트를 첨부한다. 지표는 정답이 아니라 변경 영향을 확인하는 가드레일로 읽는다(예: 알림 수 급증, 구간 평균 길이 급변).

### 8.4 가중치 영향 추적 파이프라인

`scoring.yml` 상수 변경의 영향을 운영 DB 이력 재계산으로 자동 산출한다. DB 이전 완료 전까지는 고정 경기 세트를 수동 재생해 대체하며, 이전 후 이 파이프라인으로 일원화한다. 구현은 이 문서 범위 밖(설계만)이다.

| 구분 | 내용 |
|---|---|
| 입력 | 기준 상수(baseline `version` 또는 `baseline.json`), 후보 `scoring.yml`, 대상 경기 범위(기간 또는 `game_id` 집합), `source` 필터 |
| 처리 | (1) 운영 DB에서 대상 경기의 `plays`·`games`·`standings`·`odds_snapshots`를 **읽기 전용**으로 로드 → (2) 기준·후보 두 상수 세트로 각각 scorer를 재계산한다(운영 테이블에 쓰지 않고 임시 영역·메모리에서 수행) → (3) 두 결과의 검증 지표와 아래 리포트 항목을 비교 |
| 출력 | JSON + Markdown 영향 리포트를 파일로 생성해 `tune:` PR에 첨부 |
| 실행 | 배치 프로파일·Gradle 태스크로 실행한다(읽기 전용 DB 접속). 예: `./gradlew backtestImpact -Pbaseline=<version> -Pcandidate=scoring.yml -Pfrom=<date> -Pto=<date>` |

**리포트 항목**

| 항목 | 산출 |
|---|---|
| 순위 변화 | 경기 peak `watch_score` 랭킹의 기준↔후보 순위 상관(Spearman·Kendall)과 상위 N 진입·이탈 목록 |
| 알림 빈도 | 임계 85 기준 하루 알림 수 분포(평균·최대)와 경기당 발화 수의 기준↔후보 변화 |
| 다시보기 구간 커버리지 | 구간 수, 평균 길이, 득점·리드 변경 play 포함률의 변화 |
| 정렬 품질·예측력 | 순위 상관·AUC의 기준↔후보 델타 |
| 경보 플래그 | 알림 수 급증, 구간 평균 길이 급변 등 임계 초과 항목을 강조 표시 |

## 9. 제외 항목

| 항목 | 사유 |
|---|---|
| 상위 N개 경기만 정밀 수집 | 호출량이 동시 경기 수(최대 15경기 안팎)로 고정되어 제한이 불필요하고, 랭킹 자기강화가 발생한다. |
| 경기 종료 시 재분석 (구간 재계산·신뢰도 보강) | 라이브 분석 결과를 그대로 사용한다. 종료 시에는 상태 전환만 수행해 운영 경로를 단순화한다. |
| 타석 복합 점수 (득점위기·타구품질·승부처·투수상태 가중 합성) | 검증 불가능한 상수가 과다하고 사용자에게 설명하기 어렵다. 해당 신호는 이진 태그로만 사용한다. |
| 라인업 타자 평가·핵심 타자 출전율 점수 | 타순이 경기 직전에야 공개되어 예정 목록 시간대에 계산할 수 없고, 공개 시점에 점수가 급변해 정렬이 불안정해진다. |
| 최근 10경기·연승연패 점수 | 시즌 성적과 중복되고 변동성만 크다. |
| 선발 매치업 5지표 가중합 (ERA·WHIP·WAR·K/9·QS율) | ERA 단일 지표 대비 판별 이득 없이 상수만 늘어난다. |
| 라이브 배당 신호 | 유불리 방향을 내포해 스포일러 프리 원칙과 충돌하고, 갱신(15~20분)이 라이브 신호보다 늦다. |
| 오프닝 배당 | 2026 시즌 미제공. 당일 첫 관측 스냅샷으로 대체한다. |
| `fielding_fip` | 실측값이 통상 범위를 벗어난다. |
| 경기 내 세부 이벤트별 알림 | 상세 화면 실시간 태그·문구로 대체한다. |
| 관심 선수 알림 | 정렬 가산·태그·상세 표시로 대체한다. |
| 선수 props, 구종별 매치업, splits/versus | 점수 재료가 아니라 문구 고도화 단계의 소재다. |
