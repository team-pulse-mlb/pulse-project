# 사용자 추천 정책

이 문서는 계산된 신호를 사용자 화면, 알림, AI 문구에 어떻게 노출할지 정의한다. 점수 자체는 내부 정렬과 판단에만 사용하고, 사용자는 스포일러가 제거된 태그와 문구만 본다.

## 1. 기본 원칙

| 원칙 | 정책 |
|---|---|
| 점수 비노출 | `watch_score`, `pregame_score`, `peak_base_score`는 숫자와 등급 모두 화면에 노출하지 않는다. |
| 서버 강제 | 보호 모드 필터링은 프론트가 아니라 API 응답 생성 단계에서 적용한다. |
| 결과 차단 | 점수, 승패, 리드 팀, 득점 결과, 결말을 직접 드러내지 않는다. |
| 짧은 설명 | 사용자는 계산 근거 전체가 아니라 추천 이유 태그와 한 줄 문구를 본다. |
| AI 제한 | AI는 추천 여부를 판단하지 않고, 서버가 전달한 안전한 context로 문구만 만든다. |

## 2. 추천 이유 태그

태그는 내부 신호를 보호 모드에서 읽을 수 있는 짧은 표현으로 바꾼 값이다. 이 표가 태그 표기의 정본(단일 기준)이며, AI 문구 생성 컨텍스트에도 내부 신호명이 아니라 이 표의 보호 표기 문자열만 전달한다.

| 내부 신호 | 보호 모드 태그 | 사용 위치 |
|---|---|---|
| `late_or_extra` | 후반 긴장 구간 | 카드, 상세, 다시보기 |
| `score_gap` | 접전 흐름 | 카드, 상세, 다시보기 |
| `recent_score` | 흐름 변화 | 카드, 상세, 다시보기 |
| `lead_change` | 흐름 급변 | 카드, 상세, 다시보기 |
| `big_inning` | 한 이닝 흐름 집중 | 카드, 상세, 다시보기 |
| `pressure` | 득점권 압박 | 카드, 상세, 다시보기 |
| `count_pressure` | 승부처 카운트 | 상세 |
| `early_slugfest` | 초반 난타 흐름 | 카드, 상세 |
| `pitcher_instability` | 투수 흔들림 | 상세 |
| `hard_contact` | 장타 위험 | 상세 |
| `favorite_team` | 관심 팀 | 카드, 알림 |
| `favorite_player` | 관심 선수 | 상세 |

상세 신호 중 `pitcher_instability`, `hard_contact`, `favorite_player`는 정렬 점수에 넣지 않고 태그와 상세 화면 표시에만 사용한다.

## 3. 알림 정책

알림은 경기 단위로만 보낸다. 개별 홈런, 득점, 역전, 삼진 같은 이벤트 알림은 사용하지 않는다.

| 유형 | 조건 | 대상 | 빈도 제한 | 전달 |
|---|---|---|---|---|
| 급상승 경기 알림 | `watch_score` 85 이상 진입 **그리고** 최근 5분 내 +15 이상 상승. 발화 후 70 미만으로 내려가야 재무장(히스테리시스) | 전역(관심 팀 무관), `notify_surge_enabled`로 개인별 차단 가능 | 전체 15분 1회 | 인앱 토스트 + 알림 센터 |
| 관심 팀 경기 시작 | 관심 팀 경기의 진행 중 전환 감지 | 관심 팀(`favorite_team_ids`)이 홈·원정에 포함되고 `notify_game_start`를 켠 사용자 | 경기당 1회 | 인앱 토스트 + 알림 센터 |
| 경기 전환 안내 | 다른 경기의 `watch_score`가 현재 경기보다 20점 이상 높고 70 이상 | 상세 화면 조회 중인 사용자 | 같은 후보 경기 15분 1회 | 상세 화면 토스트 (알림 파이프라인이 아니라 상세 응답의 `switchSuggestion` 필드) |

급상승 판정은 scorer가, 경기 시작 판정은 poller가 하고, 사용자별 전달·저장은 api가 한다. 임계(85)·재무장(70)·급등 조건은 사용자별 설정이 아니라 `scoring.yml` 전역 상수다. 이벤트 스키마는 [API_CONTRACTS.md](API_CONTRACTS.md) §6.

관심 선수는 알림 조건으로 사용하지 않는다. 관심 선수 정보는 정렬 가산, 태그, 상세 화면 표시로만 제공한다.

## 4. 보호 모드 필드

보호 모드는 경기를 볼지 판단하는 데 필요한 맥락만 제공한다.

| 노출 가능 | 예시 |
|---|---|
| 매치업 | 팀 이름, 팀 약어 |
| 경기 상태 | 예정, 진행 중, 종료 |
| 진행 이닝 | 이닝 숫자 (초/말은 공격팀을 드러내므로 비노출) |
| 상황 맥락 | 아웃 카운트, 볼카운트, 득점권, 만루 |
| 추천 정보 | 추천 이유 태그, AI 한 줄 문구, 다시보기 구간 범위 |
| 표시 정보 | 경기장, 시작 시각, 관심 팀/선수 여부 |

팀 이름은 매치업 판단에 필요한 기본 정보이므로 보호 모드에서도 노출한다. 단, 어느 팀이 앞서거나 유리한지는 드러내지 않는다. 화면별 노출 필드의 전체 기준과 금지 필드 목록은 [API_CONTRACTS.md](API_CONTRACTS.md) §3이다.

## 5. 공개 모드 필드

사용자가 경기 단위 공개 전환을 선택하면 아래 정보를 제공한다.

| 공개 가능 | 예시 |
|---|---|
| 점수 | 현재 점수, 최종 점수, play 전후 점수 |
| 결과 | 승패, 득점 play, 타석 결과, 원문 중계 텍스트 |
| 방향성 | 홈/원정 우세, 리드 변경 |
| 상세 매치업 | 타자/투수 이름, 이닝 초/말, play 타임라인 |

공개 전환은 확인 다이얼로그를 거치며, 해당 경기 상세에만 적용한다.

## 6. 금지 표현

보호 모드와 알림 문구에는 아래 표현을 사용하지 않는다.

| 금지 표현 | 이유 |
|---|---|
| 홈런 발생 | 구체적인 결과를 드러낸다. |
| 역전 성공 | 어느 팀의 상황인지 추측할 수 있다. |
| 끝내기 | 경기 결말을 드러낸다. |
| 실점 위기 | 특정 팀이 불리하다는 방향을 드러낸다. |
| 홈팀/원정팀 리드 | 앞서는 팀을 드러낸다. |
| 점수 차가 좁혀짐 | 점수 흐름과 방향을 드러낸다. |

## 7. AI 문구 생성

AI 문구는 서버가 만든 안전한 context만 입력으로 사용한다. 생성 결과가 금지 표현이나 점수·승패·우세 정보를 포함하면 기본 문구로 대체한다.

| 상황 | purpose | 기본 문구 |
|---|---|---|
| 라이브 추천 | `LIVE_HEADLINE` | 지금 볼 만한 흐름이 감지됐습니다. |
| 급상승 알림 | `NOTIFICATION` | 지금 볼 만한 흐름의 경기가 감지됐습니다. |
| 관심 팀 경기 시작 알림 | `NOTIFICATION` | 관심 팀 경기가 시작됐습니다. |
| 전환 안내 | `SWITCH_SUGGESTION` | 다른 경기에서 긴장감이 높아졌습니다. |
| 다시보기 | `REPLAY_SUMMARY` | 다시 볼 만한 흐름이 이어진 구간입니다. |

문구 생성은 사용자 요청 경로가 아니라 **데이터 갱신 시점에 비동기로** 수행한다. scorer가 유의미한 변화(태그 세트 변화·추천 상태 진입·구간 확정)를 감지하면 생성을 요청하고, 데드라인 내 생성+검수 통과분만 저장한다. 룰 기반 문구를 먼저 표시하고, AI 문구가 준비되면 같은 보호 규칙을 통과한 경우에만 교체한다. 라이브 문구는 Redis 캐시, 종료 경기 문구(다시보기 요약·헤드라인)는 PostgreSQL에 영속한다. 계약은 [API_CONTRACTS.md](API_CONTRACTS.md) §5.
